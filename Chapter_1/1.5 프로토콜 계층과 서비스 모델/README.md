# 1.5 프로토콜 계층과 서비스 모델

<br/>

# 1.5.1 계층구조

계층구조는 크고 복잡한 시스템의 잘 정의된 특정 부분을 논의할 수 있게 해주며, 이러한 단순화는 매우 중요하다.

<br/>

시스템이 계층구조를 가질 때, 그 계층이 제공하는 서비스의 구현을 변경하는 것도 매우 쉽다.

어떤 한 계층의 구현이 변하더라도 시스템의 나머지 부분은 변하지 않는다는 것이다.

<br/>

> 💡 계층구조의 각 계층은 **(1) 그 계층에서 어떤 동작을 취하고 (2) 그 계층 바로 아래 계층 서비스를 사용**함으로써 서비스를 제공한다.

<br/>
<br/>

## 프로토콜 계층화

네트워크 프로토콜의 설계 구조를 제공하기 위해,  
네트워크 설계자는 프로토콜(프로토콜을 구현하는 네트워크 하드웨어와 소프트웨어)을 `계층(layer)`으로 조직한다.

즉, **각각의 프로토콜은 한 계층에 속하며**, 프로토콜 계층은 소프트웨어, 하드웨어 또는 둘의 통합으로 구현할 수 있다.

<br/>

- 한 계층은 상위 계층에 제공하는 <b>서비스(service)</b>에 관심을 갖고, 이것을 계층의 `서비스 모델(service model)`이라고 한다.
- **각 계층은 그 계층 내부에서 어떤 동작을 수행하거나, 직접 하위 계층의 서비스를 사용한다.**

즉,

- `서비스`는 **한 계층이 바로 위 계층에게 무엇을 제공하는가**에 대한 관점이고
- `프로토콜`은 **그 서비스를 실제로 구현하기 위해 같은 계층의 양 끝 개체들이 따르는 규칙**이라고 볼 수 있다.

<br/>

다양한 계층의 프로토콜을 합하여 `프로토콜 스택(protocol stack)`이라고 한다.

<br/>
<br/>

## 인터넷 프로토콜 스택

**인터넷 프로토콜 스택은 5개 계층**으로 구성된다.

> 물리, 링크, 네트워크, 트랜스포트, 애플리케이션 계층

*아래부터 1계층 ~ 5계층*

쉽게 보면, 각 계층이 다루는 전달 범위는 다음과 같이 점점 추상화된다.

- `링크 계층` : **인접한 두 노드 사이**
- `네트워크 계층` : **출발지 호스트부터 목적지 호스트까지**
- `트랜스포트 계층` : **출발지 프로세스부터 목적지 프로세스까지**

### 애플리케이션(Application) 계층

> 💡 네트워크 애플리케이션과 애플리케이션 계층 프로토콜이 있는 곳이다.

- 인터넷의 애플리케이션 계층이 포함하는 대표적 프로토콜은 다음과 같다.
    - HTTP : 웹 문서 요청과 전송 제공
    - SMTP : 전자메일 전송 제공
    - FTP : 두 종단 시스템 간의 파일 전송 제공
- `도메인 네임 서버(domain name server, DNS)`는 이 애플리케이션 계층에 존재한다.
- 애플리케이션 계층 프로토콜은 여러 종단 시스템에 분산되어 있어서  
  한 종단 시스템에 있는 애플리케이션이 다른 종단 시스템에 있는 애플리케이션과 `정보 패킷(메시지, message)`을 교환하는 데 이 프로토콜을 사용한다.

즉, 사용자가 실제로 접하는 **웹, 메일, 파일 전송 같은 네트워크 기능**이 이 계층에서 구현된다.

<br/>

### 트랜스포트(Transport) 계층

> 💡 클라이언트와 서버 간에 애플리케이션 계층 메시지를 전송하는 서비스를 제공한다.

- 트랜스포트 계층 패킷 = `세그먼트(segment)`
- 트랜스포트 프로토콜의 두 가지 종류로는 다음과 같으며, **이들은 애플리케이션 계층 메시지를 전달한다.**
    - `TCP`
        - 애플리케이션에게 **연결 지향형 서비스**를 제공한다.
        - 목적지로의 애플리케이션 계층 메시지 **전달 보장**과 <b>흐름 제어(송신자/수신자의 속도 일치)</b>를 포함한다.
        - 긴 메시지를 짧은 메시지로 나누고, **혼잡 제어 기능**을 제공한다. (네트워크가 혼잡할 때 출발지의 전송률을 줄임)
    - `UDP`
        - 애플리케이션에게 **비연결형 서비스**를 제공한다.
        - **신뢰성, 흐름 제어, 혼잡 제어를 제공하지 않는다.**

쉽게 말해, 트랜스포트 계층은 **호스트 안의 어떤 애플리케이션 프로세스가 이 데이터를 받아야 하는지까지 포함해서** 전달을 담당한다.

<br/>

### 네트워크(Network) 계층 (= IP 계층)

> 💡 한 호스트에서 다른 호스트로 `데이터그램(datagram, IP의 전송 단위)`을 라우팅하는 책임을 진다.

즉, 출발지와 목적지 간 일련의 패킷 스위치(인터넷에서는 라우터)를 통해 데이터그램을 라우트한다.

1. 출발지 호스트에서 인터넷 트랜스포트 계층 프로토콜(TCP 또는 UDP)은 **트랜스포트 계층 세그먼트와 목적지 주소를 네트워크 계층으로 전달**한다.
2. 네트워크 계층은 **목적지 호스트의 트랜스포트 계층으로 세그먼트를 운반**하는 서비스를 제공한다.

<br/>

> 네트워크 계층은 IP 데이터그램의 필드를 정의하며
**종단 시스템과 라우터가 이 필드에 어떻게 동작하는지를 정의하는 프로토콜,** `IP 프로토콜`을 가지고 있다.

- 인터넷의 네트워크 계층에서 공통적으로 사용되는 핵심 프로토콜은 `IP`이다.
- 따라서 인터넷에 연결된 호스트와 라우터는 모두 IP를 이해하고 처리할 수 있어야 한다.

<br/>

> 인터넷 네트워크 계층은 **라우팅 프로토콜**을 포함한다.

- `라우팅 프로토콜`은 출발지와 목적지 사이에서 데이터그램이 이동하는 경로를 결정한다.
- 인터넷은 네트워크의 네트워크이며, 한 네트워크 내부에서 네트워크 운용자는 **원하는 어떠한 라우팅 프로토콜이라도 수행할 수 있다.**

<br/>

### 링크(Link) 계층

> 💡 전체 프레임을 한 네트워크 요소에서 이웃 네트워크 요소로 이동시킨다.

링크 계층 패킷 = `프레임(frame)`

<br/>

> 경로상의 **한 노드(호스트 혹은 패킷 스위치)에서 다른 노드로 패킷을 이동하기 위해** 네트워크 계층은 링크 계층 서비스에 의존해야 한다.

1. 각 노드에서 네트워크 계층은 데이터그램을 아래 링크 계층으로 보내고
   **링크 계층은 그 데이터그램을 경로상의 다음 노드에 전달한다.**
2. 다음 노드에서 링크 계층은 그 데이터그램을 상위 네트워크 계층으로 보낸다.

<br/>

링크 계층에서 제공하는 서비스는 그 링크에서 채용된 특정 링크 계층 프로토콜에 의해 결정된다.

e.g., 어떤 프로토콜은 송신 노드로부터 하나의 링크를 통해 반대편에 있는 수신 노드까지 신뢰적인 전송을 제공한다. *(이는 TCP의 신뢰적인 전달 서비스와는 다름)*

데이터그램은 출발지에서 목적지로 가는데 여러 링크를 거치므로, **데이터그램은 경로상의 각기 다른 링크에서 다른 링크 계층 프로토콜에 의해 처리될 수 있다.**

즉, 링크 계층은 **끝까지 한 번에 보내는 계층이 아니라, "바로 다음 장비까지" 넘기는 계층**이라고 이해하면 된다.

<br/>

### 물리(Physical) 계층

> 💡 프레임 내부의 각 비트를 한 노드에서 다음 노드로 이동시킨다.

이 계층의 프로토콜들은 링크에 의존하고, 더 나아가 링크의 실제 전송 매체(꼬임쌍선, 단일 모드 광케이블 등)에 의존한다.

<br/>

### 실제 애플리케이션 예시

예를 들어, 사용자가 브라우저에서 `https://www.example.com`에 접속해 웹페이지를 받아오는 상황을 생각해보자.

사용자는 단순히 주소창에 URL을 입력했을 뿐이지만, 실제 내부에서는 아래와 같은 일이 순서대로 일어난다.

#### 1. 사용자가 URL을 입력한다.

- 사용자가 브라우저 주소창에 `https://www.example.com`을 입력한다.
- 브라우저는 먼저 `www.example.com`에 접속하려면 **어느 IP 주소로 가야 하는지** 알아야 한다.

#### 2. 애플리케이션 계층에서 DNS 질의가 일어난다.

- 브라우저는 운영체제의 리졸버(resolver)에 `www.example.com`의 IP 주소를 요청한다.
- 로컬 캐시에 없으면, DNS 서버에 질의 메시지를 보낸다.
- 이 DNS 질의와 응답은 `DNS 프로토콜`을 사용하며, DNS는 **애플리케이션 계층**에 속한다.
- 예를 들어 DNS 응답으로 `www.example.com -> 93.184.216.34` 같은 IP 주소를 얻게 된다.

즉, 이 단계는 "이름을 IP 주소로 바꾸는 단계"다.

#### 3. 애플리케이션 계층에서 HTTP 요청을 준비한다.

- 이제 브라우저는 웹 서버의 IP 주소를 알게 되었으므로, 서버에 보낼 `HTTP 요청 메시지`를 만든다.
- 예를 들어 `"GET / HTTP/1.1"` 같은 요청 라인과 여러 헤더가 만들어진다.
- HTTPS라면 실제로는 HTTP 전에 TLS 관련 절차도 추가되지만, 여기서는 큰 흐름만 본다.

즉, 이 단계는 "무엇을 요청할지 메시지를 만드는 단계"다.

#### 4. 트랜스포트 계층이 서버 프로세스까지 전달할 준비를 한다.

- 브라우저가 만든 HTTP 요청 메시지는 `트랜스포트 계층`으로 내려간다.
- 웹 접속은 보통 `TCP`를 사용하므로, 먼저 서버와 TCP 연결을 설정한다.
- 그 다음 HTTP 메시지는 TCP 세그먼트로 나뉘어 전달된다.
- TCP는 다음과 같은 정보를 함께 다룬다.
  - 출발지 포트 번호
  - 목적지 포트 번호 (예: HTTP는 80, HTTPS는 443)
  - 순서 번호
  - 오류 제어와 재전송을 위한 정보

즉, 트랜스포트 계층은 단순히 "상대 호스트"까지만 보내는 것이 아니라, **그 호스트 안의 어떤 애플리케이션 프로세스가 이 데이터를 받아야 하는지까지 포함해서** 전달한다.

#### 5. 네트워크 계층이 목적지 호스트까지 경로를 잡는다.

- TCP 세그먼트는 `IP 데이터그램`에 담긴다.
- 이때 IP 헤더에는 출발지 IP 주소와 목적지 IP 주소가 들어간다.
- 데이터그램은 여러 라우터를 거쳐 목적지 호스트까지 이동한다.
- 중간 라우터들은 TCP나 HTTP 내용을 보는 것이 아니라, 주로 **IP 주소를 보고 다음 홉(next hop)** 을 결정한다.

즉, 네트워크 계층은 "이 데이터를 어떤 호스트까지 어떤 경로로 보낼까?"를 담당한다.

#### 6. 링크 계층이 한 홉씩 다음 장비로 넘긴다.

- 현재 노드는 IP 데이터그램을 바로 멀리 있는 목적지까지 한 번에 보내는 것이 아니라, **현재 연결된 다음 장비**로 보낸다.
- 그래서 각 홉에서 데이터그램은 링크 계층의 `프레임`에 실린다.
- 예를 들어,
  - 내 PC -> 가정용 공유기
  - 공유기 -> 통신사 라우터
  - 통신사 라우터 -> 또 다른 라우터
  - ... -> 목적지 서버
- 패킷이 라우터를 하나 지날 때마다, 링크 계층 프레임은 그 링크에 맞는 형태로 다시 만들어질 수 있다.

즉, 링크 계층은 **끝까지 보내는 계층이 아니라, 바로 다음 장비까지 넘기는 계층**이다.

#### 7. 물리 계층이 실제 신호를 보낸다.

- 프레임 안의 비트들은 전기 신호, 광 신호, 무선 신호 등의 형태로 실제 매체를 따라 이동한다.
- 우리가 흔히 "인터넷으로 데이터가 간다"고 말할 때, 실제 맨 아래에서는 이런 물리적 신호 전송이 일어나고 있다.

#### 8. 수신 측에서는 역순으로 벗겨진다.

- 서버에 도착한 비트는 물리 계층 -> 링크 계층 -> 네트워크 계층 -> 트랜스포트 계층 -> 애플리케이션 계층 순으로 올라간다.
- 링크 계층은 프레임을 해석하고,
- 네트워크 계층은 IP 헤더를 보고,
- 트랜스포트 계층은 TCP 헤더와 포트 번호를 보고,
- 최종적으로 웹 서버 프로세스가 HTTP 요청 메시지를 받는다.

즉, 송신 측에서는 캡슐화가 일어나고, 수신 측에서는 역캡슐화가 일어난다.

#### 9. 서버의 응답도 같은 구조로 돌아온다.

- 서버는 요청을 처리한 뒤 HTML, CSS, JavaScript 등의 데이터를 담은 `HTTP 응답 메시지`를 만든다.
- 이 응답도 다시 TCP 세그먼트 -> IP 데이터그램 -> 링크 계층 프레임 -> 물리 신호의 형태로 전달된다.
- 브라우저는 응답을 받아 웹페이지를 화면에 렌더링한다.

정리하면, 웹페이지 하나를 여는 과정에서도 실제로는 아래가 모두 함께 동작한다.

- `애플리케이션 계층` : DNS, HTTP
- `트랜스포트 계층` : TCP
- `네트워크 계층` : IP
- `링크 계층` : 각 홉 사이 프레임 전달
- `물리 계층` : 실제 비트 신호 전송

### 라우터와 링크 계층 스위치

- 이들은 둘 다 `패킷 교환기`다.
- 종단 시스템과 비슷하게, 라우터와 링크 계층 스위치는 네트워킹 하드웨어와 소프트웨어를 계층으로 구성한다.
- 그러나 모든 계층을 구현하지는 않고, **일반적으로 하위 계층을 구현한다.**  
  (그림에서는 링크 계층 스위치가 1, 2 계층을 구현하고 라우터는 1~3 계층을 구현)
- 즉, 인터넷 라우터들은 IP 프로토콜(3계층 프로토콜)을 구현할 수 있지만, 링크 계층 스위치는 불가하다.

쉽게 구분하면 다음과 같다.

- `링크 계층 스위치` : **MAC 주소와 프레임**을 보고 같은 링크 계층 네트워크 안에서 전달
- `라우터` : **IP 주소와 데이터그램**을 보고 서로 다른 네트워크 사이에서 전달

<br/>

### 호스트

이는 다섯 계층 모두를 구현한다.

> 💡 인터넷 구조가 네트워크의 ‘가장자리’에서 그 복잡성을 유지한다.

## 캡슐화(encapsulation)

> 💡 각 계층에서 패킷은 `헤더 필드`와 `페이로드 필드(payload field)`라는 두 가지 형태의 필드를 갖는다.

페이로드(payload)는 일반적으로 그 계층 상위로부터의 패킷을 말한다.

<br/>

### 캡슐화 과정

1. 송신 호스트에서 `애플리케이션 계층 메시지(application-layer message, 위 그림에서의 M)`는 트랜스포트 계층으로 보내진다.
2. 가장 간단한 경우, 트랜스포트 계층은 메시지에 수신 측 트랜스포트 계층에서 사용될 추가 정보인 `트랜스포트 계층 헤더 정보(Ht)`를 더한다.

<br/>

> `트랜스포트 계층 세그먼트(transport-layer segment)` = 애플리케이션 계층 메시지 + 트랜스포트 계층 헤더 정보

- 트랜스포트 계층 세그먼트는 애플리케이션 계층 메시지를 **캡슐화**한다.
- 트랜스포트 계층 헤더 정보가 포함하는 내용은 다음과 같다.
    - 수신 측의 트랜스포트 계층이 그 메시지를 적절한 애플리케이션으로 보내도록 하는 정보들
    - 메시지의 비트들이 변경되었는지 아닌지를 수신자가 결정하게 하는 오류 검출 비트

<br/>

3. 트랜스포트 계층은 세그먼트를 네트워크 계층으로 보낸다.
4. 네트워크 계층은 출발지와 목적지 종단 시스템 주소와 동일한 헤더 정보(Hn)를 추가하여 `네트워크 계층 데이터그램(network-layer datagram)`을 만든다.

<br/>

5. 데이터그램은 링크 계층으로 전달된다.
6. 링크 계층도 자신의 헤더 정보를 추가하여 `링크 계층 프레임(link-layer frame)`을 만든다.

<br/>

즉, 데이터를 내려보낼수록 **원래 데이터 바깥에 각 계층의 제어 정보(헤더)가 차례대로 덧붙는다.**

### 실제로 어떻게 포장되는가?

예를 들어 브라우저가 아래와 같은 HTTP 요청을 보낸다고 해보자.

```text
GET / HTTP/1.1
Host: www.example.com
```

이 메시지가 아래 계층으로 내려가면서 다음처럼 포장된다고 생각하면 된다.

1. `애플리케이션 계층 메시지`

```text
[HTTP 메시지]
GET / HTTP/1.1
Host: www.example.com
```

2. `트랜스포트 계층 세그먼트`
   - TCP는 애플리케이션 메시지 앞에 자신의 헤더를 붙인다.
   - 예를 들어 다음 같은 정보가 들어갈 수 있다.
     - 출발지 포트: `51514`
     - 목적지 포트: `443`
     - 순서 번호
     - 체크섬

```text
[TCP 헤더 | HTTP 메시지]
[src port=51514, dst port=443, seq=..., checksum=... | GET / HTTP/1.1 ...]
```

3. `네트워크 계층 데이터그램`
   - IP는 TCP 세그먼트 앞에 IP 헤더를 붙인다.
   - 예를 들어 다음 같은 정보가 들어간다.
     - 출발지 IP: `192.168.0.10`
     - 목적지 IP: `93.184.216.34`
     - TTL
     - 프로토콜 종류(TCP인지 UDP인지)

```text
[IP 헤더 | TCP 헤더 | HTTP 메시지]
[src IP=192.168.0.10, dst IP=93.184.216.34, TTL=64, protocol=TCP | ...]
```

4. `링크 계층 프레임`
   - 링크 계층은 다시 그 앞에 링크 계층 헤더를 붙인다.
   - 예를 들어 이더넷이라면 다음 같은 정보가 들어간다.
     - 출발지 MAC 주소
     - 목적지 MAC 주소
     - 타입(IP 패킷인지 등)

```text
[Ethernet 헤더 | IP 헤더 | TCP 헤더 | HTTP 메시지]
[src MAC=AA:AA:AA:AA:AA:AA, dst MAC=BB:BB:BB:BB:BB:BB, type=IP | ...]
```

5. `물리 계층`
   - 마지막으로 이 프레임 안의 비트들이 전기/광/무선 신호로 바뀌어 실제 선로를 통해 전송된다.

중요한 점은 다음과 같다.

- `HTTP 메시지` 자체는 애플리케이션이 만든 원본 데이터다.
- `TCP 헤더`는 **어느 프로세스에 보내야 하는지**, 순서가 맞는지 등을 위해 붙는다.
- `IP 헤더`는 **어느 호스트까지 가야 하는지**를 알리기 위해 붙는다.
- `링크 계층 헤더`는 **지금 이 링크에서 바로 다음 장비가 누구인지**를 알리기 위해 붙는다.

즉, 같은 데이터라도 계층이 내려갈수록

```text
메시지 -> 세그먼트 -> 데이터그램 -> 프레임
```

처럼 이름이 바뀌고, 실제로는 **앞부분에 헤더가 계속 덧붙는 구조**라고 이해하면 된다.

### 역캡슐화(decapsulation)

수신 측에서는 반대로 아래 계층부터 위 계층으로 올라가며 각 계층이 자신이 붙인 헤더를 해석하고 제거한다.

즉,

1. 링크 계층은 프레임의 링크 계층 헤더를 처리한 뒤 데이터그램을 네트워크 계층으로 넘긴다.
2. 네트워크 계층은 IP 헤더를 처리한 뒤 세그먼트를 트랜스포트 계층으로 넘긴다.
3. 트랜스포트 계층은 트랜스포트 헤더를 처리한 뒤 원래 애플리케이션 메시지를 해당 애플리케이션에 전달한다.

그래서 `캡슐화`는 송신 측에서 헤더를 **붙이는 과정**, `역캡슐화`는 수신 측에서 헤더를 **벗기며 올라가는 과정**이라고 이해하면 된다.

_캡슐화 과정은 위에서 말한 것보다 더 복잡할 수 있다._

큰 메시지는 여러 개의 트랜스포트 계층 세그먼트로 분할될 수 있다.  
또한 상황에 따라 네트워크 계층에서도 데이터그램 단편화(fragmentation)가 일어날 수 있다.

**그러고 나서 수신 측에서는 필요한 재조립 과정을 거쳐 원래 데이터가 복원된다.**
